## 1. Look-alike модель для проспектов

> Суть: На основе характеристик лучших привлечённых проспектов (ставших клиентами) искать похожих.
> Агрегаты: Профиль ОКВЭД, регион, выручка, модельный сегмент; scoring новых проспектов по похожести.

**Статус: ⚠️ Частично**

| Требование | Статус | Где реализовано |
|-----------|--------|----------------|
| Вычисление характеристик клиентов | ✅ | `analysis.compute_behavioral_features()` — 5 поведенческих метрик |
| Определение «лучших» клиентов | ✅ | Топ-10% по суммарному обороту (`LOOKALIKE_TOP_DECILE=0.1`) |
| Scoring по похожести | ✅ | `analysis.compute_lookalike_scores()` — Евклидово расстояние до центроида лучших клиентов, score в диапазоне 0–1 |
| Визуализация | ✅ | `notebooks/06_behavioral_segmentation.ipynb` — топ-20 проспектов, распределение score |
| Профиль по ОКВЭД | ⚠️ | ОКВЭД как атрибут узла есть, но в lookalike-признаки **не включён** |
| Профиль по региону | ⚠️ | Регион как атрибут узла есть, но в lookalike-признаки **не включён** |
| Выручка как признак | ❌ | Выручка клиента **не извлекается** из Hive (не предусмотрена в ETL) |
| Модельный сегмент | ❌ | Внешний модельный сегмент банка **не подключён** |

**Что реально есть**: lookalike по 5 поведенческим признакам транзакций (частота, оборот, направление, рост контрагентов, доля новых). ОКВЭД и регион — в графе, но не в признаках скоринга.

## 3. Сегментация по поведению

> Суть: Паттерн платежей (частота, объём, направление) = более точная сегментация.
> Агрегаты: Кластеризация по (кол-во месяцев, SUM, AVG, направление, кол-во контрагентов); пересечение с модельным сегментом.

**Статус: ⚠️ Частично**

| Требование | Статус | Где реализовано |

| Приоритизация для менеджера | ⚠️ | Lookalike score есть, но выдача «задач для менеджера» не реализована |

## 4. Идентификация ключевых хабов в сети

> Суть: Клиенты с максимальным числом связей — «хабы», потеря которых обрушит сеть.
> Агрегаты: COUNT(уникальных контрагентов), betweenness centrality, SUM(транзакций).

**Статус: ✅ Реализовано**

| Требование | Статус | Где реализовано |
|-----------|--------|----------------|
| COUNT уникальных контрагентов | ✅ | `unique_counterparty_count` в `compute_extended_metrics()` |
| Betweenness centrality | ✅ | `compute_centrality()` — betweenness для всех узлов |
| SUM транзакций (оборот) | ✅ | `total_in_flow`, `total_out_flow` на узле |
| Флаг хаба | ✅ | `hub_flag = True` если unique_counterparty_count > 2 × медиана |
| Ограничение связей хаба | ✅ | `hub_filter()` в `filters.py` — cap = min(max(20, √degree), 50), приоритет по `edge_score` |
| PageRank | ✅ | `compute_centrality()` |

## 11. Построение графа транзакций (метрики)

### 11.1 Базовый вес ребра

> Оборот за период, кол-во транзакций, кол-во месяцев, средний чек, доля контрагента в обороте.

**Статус: ✅ Реализовано**

| Метрика | Статус | Где |
|---------|--------|-----|
| Оборот (total_amount) | ✅ | `build_graph()` — агрегация по (source, target) |
| Кол-во транзакций (tx_count) | ✅ | `build_graph()` |
| Кол-во месяцев (active_months) | ✅ | `compute_extended_metrics()` |
| Средний чек (avg_amount) | ✅ | `build_graph()` |
| Доля контрагента в обороте (share_of_turnover) | ✅ | `compute_edge_metrics()` |

### 11.2 Важность узла

> Уникальные контрагенты, оборот, топ-k концентрация, активные месяцы.

**Статус: ✅ Реализовано**

| Метрика | Статус | Где |
|---------|--------|-----|
| Кол-во уникальных контрагентов | ✅ | `compute_extended_metrics()` → `unique_counterparty_count` |
| Суммарный оборот по всем рёбрам | ✅ | `total_in_flow`, `total_out_flow` на узле |
| Концентрация топ-k контрагентов | ✅ | `top_k_concentration` (топ-5 исходящих / суммарный, `TOP_K_COUNTERPARTIES=5`) |
| Кол-во активных месяцев | ✅ | `active_months` |

### 11.3 Комбинированный вес ребра (edge_score)

> Значимость для обеих сторон + важность узлов + стабильность во времени.

**Статус: ✅ Реализовано**

Формула в `compute_edge_score()`:

```
edge_score = 0.30 × norm_amount
           + 0.30 × bilateral_share  (доля ребра у source × доля у target)
           + 0.20 × node_importance  (avg PageRank обоих узлов)
           + 0.20 × stability        (min/max active_months)
```

Веса настраиваются через `config.py` (`EDGE_SCORE_W_*`).

### 11.4 Выделение «основных связей»

**Статус: ✅ Реализовано**

| Механизм | Статус | Где |
|---------|--------|-----|
| Порог по edge_score (top 10–20%) | ✅ | Disparity filter (`filters.disparity_filter()`) |
| Минимум рёбер на узел | ✅ | `hub_filter()` — cap_min=20 рёбер гарантировано |
| Ограничение хабов (топ-N по score) | ✅ | `hub_filter()` — cap = min(max(20, √degree), 50), оставляет топ по `edge_score` |
| Сохранение взаимных рёбер хабов | ✅ | Reciprocal tx-рёбра защищены от удаления |

### 11.5 Оперативное включение новых связей

**Статус: ❌ Не реализовано**

| Требование | Статус | Примечание |
|-----------|--------|-----------|
| Инкрементальное добавление рёбер | ❌ | ETL всегда перезаписывает данные (`mode='overwrite'`) |
| Потоковая/near-real-time обработка | ❌ | Архитектура batch-only |
| Версионирование графа | ❌ | Нет механизма сравнения граф(t) и граф(t+1) |

## Ключевые пробелы

### Данные (не подключены к ETL)
- **Выручка** клиентов — нет в `paymentcounteragent_stran`, нужна отдельная таблица
- **ОКВЭД** — атрибут в графе есть (`DEFAULT_OKVED_CODE="00"`), но реального извлечения из Hive **нет**
- **Регион** — аналогично ОКВЭД
- **Модельный сегмент банка** — внешние данные не подключены
- **Продуктовая глубина** (какие продукты клиент использует) — нет

### Функциональность
- **Динамика** (сравнение периодов) — архитектура batch за один период
- **Триггеры апсейла** — нет пороговых правил и механизма уведомления
- **Инкрементальный граф** — нет поддержки near-real-time
